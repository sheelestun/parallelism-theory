# Анализ масштабируемости алгоритма умножения матрицы на вектор

## Наблюдения

### 1. Область эффективной масштабируемости (p = 1-8)
- Для обоих размеров задачи наблюдается **близкое к линейному ускорение**
- При p = 8 ускорение составляет примерно **6×** для обеих задач
- В этой области накладные расходы на параллелизацию минимальны

### 2. Область насыщения (p > 8)
- **M = 20,000**: 
  - Максимальное ускорение **~15×** достигается при p = 20
  - При дальнейшем увеличении до p = 40 ускорение **снижается до ~13×** - перенасыщение
  
- **M = 40,000**:
  - Ускорение продолжает расти до **~14.5×** при p = 40
  - Более плавная деградация эффективности

### 3. Эффект размера задачи
При увеличении размера задачи с 20,000 до 40,000:
- **Улучшается масштабируемость** на большом числе потоков
- Меньше влияние накладных расходов параллелизации

## Причины отклонения от линейной масштабируемости

1. **Накладные расходы OpenMP**: создание потоков, синхронизация, распределение работы
2. **Конкуренция за ресурсы памяти**: при увеличении числа потоков растет конкуренция за пропускную способность памяти


## Выводы

1. **Оптимальное число потоков**: 
   - Для M = 20,000: **16-20 потоков**
   - Для M = 40,000: **до 40 потоков**

2. **Эффективность параллелизации**:
   - При p = 40 эффективность составляет **~35-36%** от идеальной
   - Реальное ускорение в 3-4 раза ниже линейного


## Метрики эффективности

| Размер | p | Ускорение | Эффективность |
|--------|---|-----------|---------------|
| 20,000 | 8 | ~6× | 75% |
| 20,000 | 20 | ~15× | 75% |
| 20,000 | 40 | ~13× | 32.5% |
| 40,000 | 40 | ~14.5× | 36.25% |

**Эффективность** = (Ускорение / p) × 100%
